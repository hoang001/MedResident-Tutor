# Database

PostgreSQL 16 with the pgvector extension is the authoritative runtime database. UUID primary keys are generated by the application. Foreign keys encode ownership and deletion behavior; lookup and ordering indexes cover common access paths.

## Initial migration

`apps/backend/alembic/versions/20260714_0001_initial_schema.py` enables `vector` and creates the baseline metadata. Run it with `alembic upgrade head`. All later schema changes must be separate reviewed Alembic revisions; never use application startup `create_all` against a deployed database.

## Tables

| Area | Tables |
|---|---|
| Identity | `users` |
| Taxonomy | `specialties`, `topics` |
| Documents | `documents`, `document_sections`, `document_chunks` |
| Lightweight KG | `concepts`, `concept_relations`, `document_concepts` |
| Questions | `questions`, `question_options`, `rubrics`, `rubric_criteria` |
| Exams | `exams`, `exam_questions`, `exam_attempts`, `user_answers` |
| Learning | `learning_results`, `recommendations` |

`document_chunks.embedding` is a dimensionless pgvector value and nullable. No approximate-nearest-neighbor index exists yet because dimension, distance metric, corpus size, and embedding provider have not been selected. `metadata` JSON fields store flexible source/grading attributes, not core relational facts.

The concept tables form an MVP directed graph in PostgreSQL with relation types `PREREQUISITE_OF`, `RELATED_TO`, and `PART_OF`. They do not implement graph traversal, GraphRAG, or automatic concept extraction.

## Test database

API tests use SQLite and `Base.metadata` to keep tests self-contained. This validates service/API behavior but cannot validate PostgreSQL extension installation or dialect-specific migration behavior. CI should add a PostgreSQL/pgvector migration job before deployment.
